## 1. 브랜치 전략

오토박스는 **Git Flow** 기반의 브랜치 전략을 사용합니다.

### 1.1 기본 브랜치 구조

```
main (master)          ← 최종 배포 (직접 작업 금지)
  │
  └── dev       ← 개발 통합 브랜치
        │
        ├── fe/기능명   ← 프론트엔드 기능 브랜치
        ├── be/기능명   ← 백엔드 기능 브랜치
        ├── ai/기능명   ← AI 기능 브랜치
        └── em/기능명   ← 임베디드 기능 브랜치
```

| 브랜치 | 용도 |
|--------|------|
| `main` (또는 `master`) | 최종 배포 기준 (직접 작업 금지) |
| `develop` | 모든 기능이 병합되는 통합 개발 브랜치 |
| `fe/*`, `be/*`, `ai/*`, `em/*` | 각 파트별 기능 브랜치 |

### 1.2 기능 브랜치 규칙

**형식:** `파트/기능명` 또는 `파트/타입-기능명`

```bash
# 예시
fe/login-page          # 프론트엔드 로그인 페이지
fe/fix-header-bug      # 프론트엔드 헤더 버그 수정
be/auth-api            # 백엔드 인증 API
be/fix-token-refresh   # 백엔드 토큰 갱신 수정
ai/image-classification # AI 이미지 분류
em/motor-control       # 임베디드 모터 제어
em/mqtt-bridge         # 임베디드 MQTT 브릿지
```

| 파트 접두사 | 설명 |
|-------------|------|
| `fe/` | Frontend (프론트엔드) |
| `be/` | Backend (백엔드) |
| `ai/` | AI |
| `em/` | Embedded (임베디드) |

> 모든 기능 브랜치는 **develop 브랜치에서 분기**합니다.

## 2. 개발 환경 세팅

### 2.1 저장소 클론 및 브랜치 설정

#### Step 1: 저장소 클론

```bash
git clone <REPO_URL>
cd <REPO_NAME>
```

#### Step 2: develop 브랜치 체크아웃

```bash
git checkout develop
git pull origin develop
```

#### Step 3: 기능 브랜치 생성

```bash
# 프론트엔드 기능 개발
git checkout -b fe/login-page

# 백엔드 기능 개발
git checkout -b be/auth-api

# AI 기능 개발
git checkout -b ai/object-detection

# 임베디드 기능 개발
git checkout -b em/sensor-control
```

### 2.2 파트별 의존성 설치

#### Frontend (폴더명: `frontend`)

```bash
cd frontend
npm install
npm run dev
```

#### Backend (폴더명: `backend`)

```bash
cd backend
python -m venv .venv
source .venv/bin/activate  # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

#### AI (폴더명: `ai`)

```bash
cd ai
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

#### Embedded (폴더명: `embedded`)

```bash
cd embedded/raspberry-pi
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

## 3. 개발 워크플로우

### 3.1 전체 흐름도

```
1. develop 브랜치 최신화
      ↓
2. 기능 브랜치 생성 (fe/xxx, be/xxx 등)
      ↓
3. 개발 & 커밋
      ↓
4. 기능 브랜치 푸시
      ↓
5. MR(PR) 생성 (기능 브랜치 → develop)
      ↓
6. 코드 리뷰 & Approve
      ↓
7. 머지 완료 → 기능 브랜치 삭제
```

### 3.2 상세 단계

#### Step 1: 최신 코드 동기화

```bash
git checkout develop
git pull origin develop
```

#### Step 2: 기능 브랜치 생성

```bash
git checkout -b fe/login-page
```

#### Step 3: 개발 & 커밋

```bash
# 변경사항 확인
git status

# 스테이징
git add .

# 커밋 (컨벤션 준수)
git commit -m "[FE] feat: 로그인 페이지 구현"
```

#### Step 4: 기능 브랜치 푸시

```bash
git push origin fe/login-page
```

#### Step 5: MR(PR) 생성

- GitLab/GitHub에서 MR(PR) 생성
- 대상: `fe/login-page` → `develop`

#### Step 6: 머지 후 정리

```bash
# develop 최신화
git checkout develop
git pull origin develop

# 로컬 기능 브랜치 삭제
git branch -d fe/login-page
```

## 4. 커밋 컨벤션

### 4.1 커밋 메시지 형식

```
[파트] type: 설명
```

### 4.2 파트 코드

| 파트 코드 | 설명 |
|-----------|------|
| `FE` | Frontend |
| `BE` | Backend |
| `AI` | AI |
| `EM` | Embedded |
| `DOCS` | 문서 |
| `CHORE` | 설정/기타 |

### 4.3 Type 종류

| Type | 설명 |
|------|------|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 수정 (README 등) |
| `refactor` | 코드 리팩토링 (기능 변경 없이 코드 개선) |
| `chore` | 설정, 빌드, 패키지 등 기타 작업 |
| `style` | 코드 포맷팅, 세미콜론 누락 등 |
| `test` | 테스트 코드 추가/수정 |

### 4.4 커밋 예시

```bash
# 좋은 예시
git commit -m "[FE] feat: 로그인 페이지 구현"
git commit -m "[BE] fix: 토큰 만료 처리 수정"
git commit -m "[AI] feat: 이미지 분류 모델 추가"
git commit -m "[EM] feat: MQTT 브릿지 구현"
git commit -m "[DOCS] docs: README 업데이트"
git commit -m "[BE] refactor: 인증 로직 분리"

# 나쁜 예시
git commit -m "로그인 구현"           # 파트, type 없음
git commit -m "[FE] 로그인 구현"      # type 없음
git commit -m "feat: 로그인 구현"     # 파트 없음
```

## 5. MR(PR) 및 머지 규칙

### 5.1 MR(PR) 제목 형식

```
[파트] type: 작업 내용
```

**예시:**

```
[FE] feat: 로그인/회원가입 페이지 구현
[BE] fix: 토큰 갱신 로직 수정
[AI] feat: 객체 탐지 모델 통합
[EM] feat: MQTT 브릿지 구현
```

### 5.2 MR(PR) 병합 규칙

| 규칙 | 설명 |
|------|------|
| Approve | **최소 1명 이상** 코드 리뷰 승인 필요 |
| Conflict | 충돌 해결 후 MR 요청 |
| Self-review | MR 생성 전 본인 코드 리뷰 필수 |

### 5.3 병합 흐름

```
기능 브랜치 → develop → main
```

**예시:**

```
fe/login-page → develop → main
be/auth-api → develop → main
```

### 5.4 머지 후 브랜치 삭제 규칙

| 브랜치 유형 | 예시 | 머지 후 삭제 |
|-------------|------|--------------|
| **메인 브랜치** | `main`, `develop` | ❌ 삭제 안 함 |
| **기능 브랜치** | `fe/login`, `be/auth-api`, `em/mqtt` | ✅ 삭제 |

```bash
# 기능 브랜치 머지 후 삭제 방법

# 로컬 브랜치 삭제
git branch -d fe/login-page

# 원격 브랜치 삭제
git push origin --delete fe/login-page
```

## 6. 주의사항

### 6.1 절대 금지 사항

| 금지 | 이유 |
|------|------|
| `main`, `develop`에 직접 push | 코드 품질 관리 |
| API Key, Token 등 민감 정보 커밋 | 보안 |
| `node_modules`, `.venv`, `dist` 커밋 | 저장소 용량 |


## 7. 부록: 임베디드 전용 가이드

임베디드 장비(라즈베리파이 / Jetson)에서는 저장소 전체가 아닌 **embedded 폴더와 docs 폴더만** 필요합니다.

### 7.1 최초 설정 (한 번만)

Sparse Checkout을 사용하면 `embedded/`와 `docs/` 폴더만 클론됩니다.

```bash
# 1. 부분 클론
git clone --filter=blob:none --no-checkout https://lab.ssafy.com/s14-webmobile3-sub1/S14P11A403.git
cd S14P11A403

# 2. Sparse checkout 초기화
git sparse-checkout init --cone

# 3. embedded + docs 폴더 지정
git sparse-checkout set embedded docs

# 4. dev 브랜치 체크아웃
git checkout dev
```

**결과:**

```
S14P11A403/
├── embedded/          ← 임베디드 코드
│   └── raspberry-pi/
└── docs/              ← 문서
```

### 7.2 일반 개발 워크플로우 (매번)

최초 설정 이후에는 일반 Git 명령어로 개발합니다.

```bash
# 1. 최신 코드 받기
git checkout dev
git pull origin dev

# 2. 기능 브랜치 생성 (새 기능 시작할 때)
git checkout -b em/기능명

# 3. 개발 & 커밋
git add .
git commit -m "[EM] feat: 기능 설명"

# 4. 푸시
git push origin em/기능명

# 5. GitLab에서 MR 생성 (em/기능명 → dev)
```

### 7.3 이미 브랜치가 있을 때

```bash
# 기존 브랜치로 이동
git checkout em/기능명

# 최신 코드 받기
git pull origin em/기능명

# 개발 계속...
```

### 7.4 최초 설정 한 줄 요약

```bash
git clone --filter=blob:none --no-checkout https://lab.ssafy.com/s14-webmobile3-sub1/S14P11A403.git && cd S14P11A403 && git sparse-checkout init --cone && git sparse-checkout set embedded docs && git checkout dev
```

### 7.3 자주 하는 실수

| 실수 | 해결 |
|------|------|
| `git clone` 전체 실행 | Sparse Checkout 사용 |
| `develop`에 직접 커밋 | `em/*` 브랜치 사용 |
| `.venv`, `__pycache__` 커밋 | `.gitignore` 확인 |

## 브랜치 명령어 정리

| 명령어 | 설명 |
|--------|------|
| `git checkout <브랜치>` | 기존 브랜치로 이동 |
| `git checkout -b <브랜치>` | 새 브랜치 생성 + 이동 |
| `git branch` | 로컬 브랜치 목록 |
| `git branch -a` | 모든 브랜치 목록 (원격 포함) |
| `git branch -d <브랜치>` | 로컬 브랜치 삭제 |
| `git push origin --delete <브랜치>` | 원격 브랜치 삭제 |
